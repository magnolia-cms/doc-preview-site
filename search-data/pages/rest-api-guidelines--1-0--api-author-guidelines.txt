# API Author Guidelines

Practical guidelines for Magnolia REST API authors to comply with adopted Zally rules.

## [](#_overview)Overview

This guide helps API authors create compliant REST APIs by providing actionable guidance for each adopted rule. Guidelines are organized by where you apply them:

-   **Maven Plugin Setup** - Enable automated linting in your module

-   **OpenAPI Specification Guidelines** - Rules enforced in your `*-openapi.yaml` files

-   **Java Implementation Guidelines** - Rules requiring code-level compliance


## [](#_maven_plugin_setup)Maven Plugin Setup

See [Maven Plugin Configuration](../plugin-config/) for instructions on enabling automated API linting in your module.

| --- | --- |

| `results` , `assets` , `pages` | `result` , `asset` , `page` |

## [](#_java_implementation_guidelines)Java Implementation Guidelines

These rules require attention in your Java code, not just OpenAPI specs.

### [](#_rule_166_avoid_link_headers)Rule 166: Avoid Link Headers

Do not use HTTP Link headers for pagination or resource relations.

**How to Comply:**

Instead of Link headers, include pagination information in the response body:

```java
// Wrong - using Link headers
Response.ok(items)
    .header("Link", "</nodes?page=2>; rel=\"next\"")
    .build();

// Correct - pagination in response body
@GET
public PaginatedResponse<Node> getNodes(@QueryParam("offset") int offset,
                                        @QueryParam("limit") int limit) {
    List<Node> items = nodeService.getNodes(offset, limit);
    long total = nodeService.getTotalCount();

    return new PaginatedResponse<>(items, offset, limit, total);
}
```

### [](#_rule_176_return_problem_json_for_all_errors)Rule 176: Return Problem JSON for All Errors

Implement exception handlers that return Problem JSON format.

**How to Comply:**

```java
@Provider
public class ProblemExceptionMapper implements ExceptionMapper<WebApplicationException> {

    @Override
    public Response toResponse(WebApplicationException exception) {
        Problem problem = Problem.builder()
            .type(URI.create("https://api.magnolia-cms.com/problems/error"))
            .title(exception.getMessage())
            .status(exception.getResponse().getStatus())
            .detail(exception.getMessage())
            .build();

        return Response.status(exception.getResponse().getStatus())
            .type("application/problem+json")
            .entity(problem)
            .build();
    }
}
```

Ensure all error responses (400, 404, 500, etc.) use this format consistently.

## [](#_quick_reference)Quick Reference

### [](#_maven_setup_checklist)Maven Setup Checklist

Before starting API development:

-   Module inherits from `magnolia-parent-pom`

-   Zally Maven plugin added to module `pom.xml`

-   OpenAPI spec path configured in plugin (`<source>`)

-   Verify linting works: `mvn verify`


### [](#_openapi_checklist)OpenAPI Checklist

Before submitting your OpenAPI spec, verify:

-   OpenAPI 3.x format (Rule 101)

-   Array properties use plural names (Rule 120)

-   Enums are `type: string` (Rule 125)

-   Enum values are UPPER\_SNAKE\_CASE (Rule 240)

-   Resource paths use plural nouns (Rule 134)

-   Custom headers use Pascal-Case (Rule 132)

-   Standard HTTP status codes only (Rule 150)

-   Success and error responses defined (Rule 151)

-   No trailing slashes in paths (Rule 136)

-   Resources identified via path segments (Rule 143)

-   Appropriate nesting depth (Rules 145, 146, 147)

-   Extensible enums use `x-extensible-enum` (Rule 107)

-   Root responses are objects, not arrays (Rule 110)

-   Collection params use form-style (Rule 154)

-   Numeric types have format specified (Rule 171)

-   Standard media types used (Rule 172)

-   Problem JSON schema for errors (Rule 176)

-   No version in URI paths (Rule 115)

-   Semantic versioning in `info.version` (Rule 116)


### [](#_java_checklist)Java Checklist

Before submitting your implementation, verify:

-   No Link headers in responses (Rule 166)

-   Exception handlers return Problem JSON (Rule 176)
