# REST Design – Versioning

Describes the API versioning strategy, prohibiting URI-based versioning and mandating Semantic Versioning. The chapter explains how version changes communicate compatibility guarantees and breaking changes.

## [](#115)**MUST** do not use URI versioning \[115\]

Adopted

APIs must not include the version number in the URI path.

URI versioning leads to tight coupling between clients and specific API versions and complicates long-term evolution of APIs. Instead, APIs should evolve in a backward-compatible manner whenever possible.

**Reference:** [Zalando rule #115](https://opensource.zalando.com/restful-api-guidelines/#115)

## [](#116)**MUST** use semantic versioning \[116\]

Adopted

APIs must follow Semantic Versioning (SemVer) to communicate changes clearly.

Semantic Versioning uses the format:

`MAJOR.MINOR.PATCH`

-   **MAJOR** version increments indicate breaking changes

-   **MINOR** version increments indicate backward-compatible enhancements

-   **PATCH** version increments indicate backward-compatible bug fixes


The API version must be documented in the API specification and reflected in the API meta information.

**Reference:** [Zalando rule #116](https://opensource.zalando.com/restful-api-guidelines/#116)
