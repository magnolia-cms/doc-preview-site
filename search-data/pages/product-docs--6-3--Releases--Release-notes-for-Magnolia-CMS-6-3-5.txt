# Release notes for Magnolia CMS 6.3.5

[LTS release](#support:ROOT:index.adoc) • Delivered on February 21, 2025 • Changelog: [15689](https://magnolia-cms.atlassian.net/issues/?filter=15689)

**Magnolia 6.3.5** is primarily a bug-fixing and security release that delivers critical security fixes and [breaking changes](#_breaking_changes) in FreeMarker directives, content rendering, and in the configuration of the Content Translation Extended module.

We keep the details of the security fixes private in line with our [security policy](../../Administration/Security/Security-policy/). Contact our [Support team](#support:ROOT:index.adoc) if you need more information.

## [](#_breaking_changes)Breaking changes

The improvements and fixes shipped with this release have also introduced the following breaking changes.

### [](#_security_hardening_of_template_scripts)Security hardening of template scripts

Content is available in template scripts via templating functions and, for the most part, automatically encoded. Before this release, we identified some directives that weren’t compliant with this security encoding. To fix this, escaping content to enable encoding is now set by default for all directives.

You can run the following Groovy script to identify whether your template scripts are affected. If you’re affected, you can disable the encoding fixes by setting the new `magnolia.templating.escaping` property to `compatibility`. See [Configuration management](../../Administration/Architecture/Configuration-management/#magnolia.templating.escaping) for more details.

Alternatively, you can update the template scripts to use the new encoded content.

-   [MAGNOLIA-9563](https://magnolia-cms.atlassian.net/browse/MAGNOLIA-9563) Inconsistency in encoding content via templating functions and renderer


```groovy
import info.magnolia.objectfactory.Components;
import info.magnolia.resourceloader.ResourceOrigin;
import info.magnolia.resourceloader.util.ResourcePredicates;
import java.util.regex.Pattern;

PATTERNS = [Pattern.compile("cmsfn.(contentByPath|contentById|nodeByPath|nodeById|search|simpleSearch)"),
            Pattern.compile("searchfn.(searchPages|searchContent)"),
            Pattern.compile("catfn.(getCategories|getRelatedCategories|getContentByCategory|getCategoryNodeByName)")]

origin = Components.getComponent(ResourceOrigin.class);
resources = origin.find("/", ResourcePredicates.pathMatchesGlob("/**/*.ftl")).toList();
for (resource in resources) {
    if (searchIn(resource) > 0) {
        println resource.getPath();
    }
}


def searchIn(resource) {
    def count = 0
    resource.openReader().withCloseable { reader ->
        reader.eachLine { line ->
            PATTERNS.each{ PATTERN ->
                def matcher = PATTERN.matcher(line)
                while (matcher.find()) count++
            }
        }
    }
    return count
}
```

### [](#_content_translation_extended_module_config)Content Translation Extended module config

The release of the Content Translation Extended module `4.1.0` on January 27, 2025 entails breaking changes in the configuration of the module.

Click to see the new configuration for use with Magnolia 6.3.5.

The configuration is now located under the `translationProviders` node. Make sure that you move your existing translation provider configuration to the correct path.

```none
modules
├── content-translation-support-ext-<service-name>
├── config
├── translationProviders
│   └── <service-name>TranslationProviderConfigFile.yaml
...
```

Example for DeepL:

```none
modules
├── content-translation-support-ext-deepl
├── config
├── translationProviders
│   └── deeplTranslationProviderConfigFile.yaml
...
```
