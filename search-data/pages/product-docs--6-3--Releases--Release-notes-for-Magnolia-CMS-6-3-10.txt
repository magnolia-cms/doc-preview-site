# Release notes for Magnolia CMS 6.3.10

[LTS release](#support:ROOT:index.adoc) • Delivered on June 26, 2025 • Page updated on July 24, 2025 • Changelog: [17965](https://magnolia-cms.atlassian.net/issues/?filter=17965)

> **Warning:** Do not upgrade to Magnolia 6.3.10 from Magnolia 6.3.8 and earlier.Upgrading to Magnolia 6.3.9, 6.3.10, 6.3.11, or 6.3.12 initiates a Marketing Tags version handler task that may cause instances to become unresponsive if the repository contains a large number of versions. See the Known issues page for more information.The issue has been resolved in the Magnolia 6.3.13 release.

**Magnolia 6.3.10** enhances functionality with a new CKEditor 5 General HTML Support plugin, YAML environment variable substitution, and a separate IP security filter. Workflow dialogs now support i18n site-awareness.

This release also fixes security issues, `Content-Type` header for binaries, REST filter `OR` operator, UI responsiveness, item order preservation, and `SpaRenderer` error logs.

## [](#_improvements)Improvements

-   [MAGNOLIA-7918](https://magnolia-cms.atlassian.net/browse/MAGNOLIA-7918) Extract `ipSecurity` functionality out of `uriSecurity` filter to a separate filter

    For more, see [IP security](../../Administration/Architecture/Request-processing-and-filters/Filters/#_ip_security).

-   [MAGNOLIA-9659](https://magnolia-cms.atlassian.net/browse/MAGNOLIA-9659) Support environment variable substitution in YAML decoration files

    For more, see [Interpolate environment variables in light modules](../../Administration/Architecture/Configuration-management/#_interpolate_environment_variables_in_light_modules).

    > **Caution:** Any secrets that are substituted in YAML files may be exposed in the Definitions App. Substitutions are turned off by default, and you must be careful when setting the magnolia.yaml.envsubst feature flag to true, especially when you substitute sensitive data.

-   [MGNLCKE-40](https://magnolia-cms.atlassian.net/browse/MGNLCKE-40) Add `GeneralHtmlSupport` plugin

    For more details about the plugin and how to configure it, see [CKEditor 5 customization: General HTML Support plugin](../../Developing/Templating/Dialog-definition/Field-definition/List-of-fields/CKEditor5-customization/#_general_html_support_plugin).

-   [MGNLWORKFLOW-556](https://magnolia-cms.atlassian.net/browse/MGNLWORKFLOW-556) Show changes a dialog needs to be site-aware for i18n

#### [](#_new_allowvariants_property_in_rest_api)New `allowVariants` property in REST API

[MGNLPN-837](https://magnolia-cms.atlassian.net/browse/MGNLPN-837) introduces a new property called `magnolia.personalization.rest.allowVariants`, which enables direct retrieval of variant nodes via the REST API. The default value is `false`. For more details and recommendations, see [Retrieving variant nodes](#3.1@rest:api:Requesting-personalized-content-with-the-delivery-endpoint.adoc#_retrieving_variant_nodes).

### [](#_notable_updates)Notable updates

-   [BUILD-1025](https://magnolia-cms.atlassian.net/browse/BUILD-1025) Upgrade to SnakeYAML `2`

    There are several changes between SnakeYAML `1.x` and `2.x` which are explicitly noted as backwards-incompatible or have the potential to break existing code due to altered behavior.

    These include API changes, changes in default behavior, and security-driven restrictions. The breaking changes primarily affect:

    -   Low-level API users (for example, custom parsers or tag handling).

    -   Applications serializing JavaBeans with read-only properties.

    -   Code deserializing untrusted YAML without explicit constructor configurations.


    If you are using SnakeYAML in your project, consider the changes if you need to update. For more details, see [snakeyaml / Changes](https://bitbucket.org/snakeyaml/snakeyaml/wiki/Changes) on Bitbucket.
