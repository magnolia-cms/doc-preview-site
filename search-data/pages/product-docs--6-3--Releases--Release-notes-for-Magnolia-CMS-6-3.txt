# Release notes for Magnolia CMS 6.3.0

[LTS release](../../../../support/) • Delivered on September 4, 2024 • Changelog: [11643](https://magnolia-cms.atlassian.net/issues/?filter=11643)

**Magnolia 6.3.0** is the next step in the digital experience hub for omnichannel. In this major release, we focus particularly on the following areas:

-   Improved search, usability, and UI experience

-   Enhanced overall system performance

-   Refined flexibility of configuration


As with any Magnolia release, this release also delivers critical bug and security fixes. We keep the details of security fixes private in line with our [security policy](../../Administration/Security/Security-policy/). Contact our [Support team](../../../../support/) if you need more information.

## [](#_whats_new)What’s new

### [](#_search_experience)Search experience

We’re improving the Magnolia search experience by refining its usability and building a more efficient workflow for editors to find and manage relevant content in a seamless way.

More specifically, we bring the [in-app search for Pages and Assets apps](../../Authoring/Search/Searching-for-content/#_targeted_search_within_content_apps), together with usability enhancements to the top navigation and action bar.

![Assets app with all search filters added](../../_images/authoring/assets-app-all-filters.png)

### [](#_usability)Usability

This release brings several usability updates to the [deletion workflow](../../Authoring/Deleting-and-restoring/) as part of our ongoing efforts to improve user experience.

Users benefit from better visual feedback thanks to informative success notification pop-ups upon submitting deletion requests or completing content deletion actions.

Additionally, the Action Bar has been refined with updated titles, action labels, and icons for better clarity.

These improvements aim to make content management smoother and more intuitive for our users.

#### [](#_ckeditor_5)CKEditor 5

Magnolia 6.3 comes with version 5 of the CKEditor, a highly customizable editor which powers the Magnolia [Rich text field](../../Developing/Templating/Dialog-definition/Field-definition/List-of-fields/Rich-text-field/).

Version 5 is a modern, state-of-the-art, modular, and highly customizable editor that provides a number of [advanced formatting, editing, and productivity features](https://ckeditor.com/docs/ckeditor5/latest/features/index.html).

A full-featured rich text field with the CKEditor functions bar

![Rich text Field example](../../_images/6-2-11-lightRichTextField-richTextFieldExample.png)

> **Tip:** See also the following resources on the ckeditor.com site:Differences between CKEditor 4 and CKEditor 5Migration from CKEditor 4

### [](#_configuration)Configuration

#### [](#_microprofile_config)MicroProfile Config

Magnolia 6.3 embraces [MicroProfile Config](https://download.eclipse.org/microprofile/microprofile-config-3.0/microprofile-config-spec-3.0.html), adding state-of-the-art flexibility for external instance configuration. This is applied first to publishing receivers config (moving out of JCR).

For more details, see the following links:

-   [MicroProfile Config](../../Administration/Architecture/configuration-management/microprofile-config/)

    -   [External configuration for publishing receivers](../../Administration/Architecture/configuration-management/microprofile-config/external-configuration-for-publishing-receivers/)



##### [](#_microprofile_health)MicroProfile Health

Magnolia 6.3 provides [MicroProfile Health](https://download.eclipse.org/microprofile/microprofile-health-4.0.1/microprofile-health-spec-4.0.1.html) endpoints, facilitating deployment with liveness, readiness, and startup probes in Kubernetes environments.

Below is an example of a response payload, following the MicroProfile Health REST interface specification.

```json
{
  "status": "UP",
  "checks": [
    {
      "name": "magnolia-instance",
      "status": "UP",
      "data": {}
    }
  ]
}
```

As a consequence, the `status` path in the [Health check API](../../Developing/API/Health-check-API/#_status_path) is deprecated in Magnolia 6.3.

#### [](#_reworked_configuration_sources)Reworked configuration sources

Magnolia 6.3 streamlines the `ConfigurationSources` implementation by following the general trend of moving definitions from JCR to YAML.

**Key changes**

-   You can pull definitions from variable sources.

-   The registry keeps no resolved configuration state on its own.

-   A configuration source keeps the state of the definitions it discovers.

-   Validation can still be applied at the registry level.

-   A built-in facility for observation changes is available.


> **Caution:** A drawback of these changes is, however, that YAML configurations can no longer inherit configurations in JCR. If you want to inherit an existing definition that’s supplied via JCR configuration, you must migrate it to a YAML file.Inheriting from YAML remains fully functional.

#### [](#_default_user_groups_and_roles)Default user groups and roles

Magnolia 6.3 introduces a *modular privilege pattern* to increase the coherence and traceability of roles in the system. You can now onboard users with only an assignment to groups, with everything framed by better traceability and an overview of privileges.

Each Magnolia module is now responsible for providing roles describing access (privileges) to the resources it provides.

For more details, see [Roles, groups and users](../../Administration/IAM/Roles-groups-and-users/).

#### [](#_multiple_login_providers_for_sso)Multiple login providers for SSO

Magnolia 6.3 provides the ability to use default Magnolia users as well as SSO-based users to log into Magnolia. This guarantees that you can always access the Magnolia instance even if your Identity Provider (IdP) is down.

For more details on SSO updates, see the [master@magnolia-sso:ROOT:changelog.adoc](#master@magnolia-sso:ROOT:changelog.adoc).

#### [](#_ietf_language_tags)IETF language tags

Magnolia now supports [*IETF Language Tags*](https://en.wikipedia.org/wiki/IETF_language_tag) (also known as *BCP 47*). These tags are used in most web standards and allow for a more flexible and semantically accurate representation of language variations and local specificities.

The support is done through a more permissive `languageTag` property in [language configuration](../../Administration/Language/Language-configuration/) (`LocaleDefinitions`).

> **Note:** Since Magnolia 6.3.0, the IETF-compliant i18n support (info.magnolia.cms.i18n.IetfI18nContentSupport) is the default implementation.

## [](#_java_17)Java 17

Java 17 is the build baseline of Magnolia 6.3. Magnolia 6.3 also runs on the latest Java versions, as specified in the [Certified stack](../../Administration/Certified-stack/).

## [](#_artifact_changes)Artifact changes

### [](#_unification_of_extensions)Unification of extensions

Each module has been re-categorized into a [Core module, Extension, or Framework](../../Modules/), as well as being assigned to a defined [business-case category](../../). Modules follow a new uniform module structure, which is easier to understand for both the user and the developer.

> **Tip:** For more details, see Old module packs, new extensions.

### [](#_travel_demo_webapps_extracted_to_the_demo_project)Travel Demo webapps extracted to the Demo project

The Community and DX Core Demo webapps are now part of the Magnolia Demo project. Their artifact names have changed:

-   For DX Core, from `dx-core-demo-webapp` to `travel-demo-webapp`.

-   For Community, from `magnolia-community-demo-webapp` to `travel-demo-community-webapp`.


### [](#_local_image_recognition_module_discontinued)Local image recognition module discontinued

We are discontinuing the option to use a local, pre-trained neural network for the [Image Recognition module](#image-recognition:ROOT:index.adoc). This feature was experimental and relied on outdated and unmaintained libraries. Amazon Rekognition becomes the reference implementation for the Image Recognition module.
